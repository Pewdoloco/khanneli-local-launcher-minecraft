# Архитектура

## Стек
**.NET 8 / C#, WPF (UI позже) + консольное ядро на этапе 1.**

Почему:
- Нативный Windows, легко собирается в один exe (`PublishSingleFile`)
- Встроенный `System.Security.Cryptography` для SHA-256 — не нужна сторонняя либа для хеш-проверки
- Хорошо управляет процессами (`Process`, `StandardInput/Output`) — важно для обёртки над `.bat` и консолью сервера
- `HttpClient` + `Octokit` (опционально) для работы с GitHub Releases API
- Один язык/рантайм что для Client, что для Admin режима (Admin = Client + доп. модуль, а не отдельное приложение)

## Модули (этап 1 — Client)
```
VanillaLauncher.Client/
  Program.cs          — точка входа, консольный прогон
  ManifestModel.cs    — модель manifest.json
  ManifestService.cs  — скачивание manifest.json из GitHub Releases
  HashService.cs       — вычисление и сверка SHA-256
  UpdateService.cs     — сравнение локального состояния с манифестом, план докачки
  Downloader.cs        — докачка файлов по списку из плана
  AppConfig.cs          — пути, URL релиза, локальная папка сборки
```

## Формат manifest.json
```json
{
  "version": "26.1.2-b3",
  "generatedAt": "2026-07-12T00:00:00Z",
  "files": [
    {
      "path": "mods/sodium.jar",
      "sha256": "…",
      "size": 123456,
      "url": "https://github.com/Pewdoloco/VanillaLauncher-localServer/releases/download/26.1.2-b3/sodium.jar"
    }
  ]
}
```
- `path` — относительный путь внутри клиентской папки профиля
- `sha256` — эталонный хеш файла
- `url` — прямая ссылка на GitHub Release asset (публичный репозиторий) либо
  на API-эндпоинт asset (приватный репозиторий + токен — отдельный этап)

## Поток обновления клиента (этап 1, без UI)
1. Скачать актуальный `manifest.json` из последнего релиза
2. Пройти по локальной папке сборки, посчитать SHA-256 существующих файлов
3. Сравнить с манифестом → получить список: новые / изменённые / без изменений
4. Скачать новые/изменённые файлы во временную папку → атомарно переместить на место
5. Логировать результат (что скачано, что пропущено)

## Модули (этап 2 — Client UI)
```
VanillaLauncher.Client.UI/
  App.xaml / App.xaml.cs
  MainWindow.xaml       — статус, кнопка "Проверить обновления", кнопка "Обновить", прогресс-бар, журнал
  MainWindow.xaml.cs    — код-behind: авто-проверка при запуске, докачка через существующие сервисы Client
  appsettings.json      — Link на src/VanillaLauncher.Client/appsettings.json (один источник конфига на оба exe)
```
UI не дублирует логику — ссылается на `VanillaLauncher.Client.csproj` и использует те же `AppConfig`, `ManifestService`, `UpdateService`, `Downloader`, `HashService`.

Для прогресс-бара `Downloader.ApplyAsync` в этапе 2 сменил `IProgress<string>` на структурный `IProgress<DownloadProgress>` (файл, completed/total, стадия Started/Completed) — по нему считается доля докачанных файлов. Байтовый прогресс внутри одного файла не считается (не требовалось спецификацией).

## Модули (этап 3 — Admin)
```
VanillaLauncher.Admin/
  ServerProcessController.cs   — обёртка над start.bat: старт через cmd.exe, построчный
                                  захват stdout/stderr, штатный stop через stdin

VanillaLauncher.Client.UI/
  AdminWindow.xaml(.cs)         — Старт/Стоп, лог консоли; открывается кнопкой
                                   "Режим администратора" из MainWindow
```
`ServerProcessController` — отдельная библиотека (не зависит от Client), Admin-окно ссылается
на неё напрямую, как MainWindow ссылается на Client.

Важный нюанс реализации: `cmd.exe /c` с **относительным** именем `.bat`-файла ненадёжно
резолвится в этом окружении (`dir`/`cd` подтверждают правильный working directory, .exe
запускаются нормально, но `.bat` — нет), поэтому `ServerProcessController` всегда передаёт
**абсолютный** путь к `.bat`.

Остановка: `stop` пишется в stdin процесса, затем stdin сразу закрывается (после этой команды
серверу ввод больше не нужен). Закрытый stdin — это то, что даёт финальному `pause` в `.bat`
завершиться сразу через EOF, а не зависнуть в ожидании интерактивного ввода. Никакого
принудительного `Kill` нет — только штатная остановка с таймаутом (по умолчанию используется
в UI 60 секунд); если сервер не уложился, это отражается в UI как предупреждение.

`ServerDirectory`/`ServerBatFileName` — новые поля `AppConfig`/`appsettings.json`, опциональные
(нужны только для Admin-режима, чисто клиентская сборка их не требует).

Авторизация Admin-режима на момент Этапа 3 ещё не была реализована — сделана отдельно в Этапе 6
(см. ниже).

## Модули (этап 4 — пересоздание мира)
```
VanillaLauncher.Admin/
  WorldBackupService.cs      — BackupWorld / RecreateWorld: архивация папки мира в zip
                                (backups/<levelName>_<таймстамп>.zip) + ротация по лимиту
  ServerPropertiesReader.cs  — level-name читается из server.properties, а не задаётся
                                отдельно в appsettings.json (одна точка правды)
```
`RecreateWorld` = обязательный `BackupWorld` (правило SPEC — без бэкапа не удаляем), затем
`Directory.Delete` папки мира; сервер сгенерирует новый мир на её месте при следующем запуске.
Метод синхронный — UI сам решает выносить ли вызов в `Task.Run`, чтобы не блокировать интерфейс
во время архивации.

`WorldBackupService` не проверяет, запущен ли сервер, — эта проверка на стороне UI
(`AdminWindow` отказывает, если `ServerProcessController.IsRunning`, и просит сначала
остановить сервер). Пересоздание мира на работающем сервере не поддерживается: оно бы
писало поверх данных, которые сервер ещё держит открытыми.

Ротация — по количеству файлов `<levelName>_*.zip` в `backups/`, самые старые (по времени
создания файла) удаляются сверх `MaxBackupsToKeep` (новое поле `AppConfig`, по умолчанию 5).

`AdminWindow`: кнопка "Пересоздать мир (с бэкапом)" — отказ, если сервер запущен;
иначе нативное подтверждение (`MessageBox`, Yes/No, дефолт No) с именем мира и явным
предупреждением, что мир будет удалён; после подтверждения — бэкап и удаление, путь к
бэкапу и результат пишутся в лог.

## Модули (этап 5 — публикация одной кнопкой)
```
VanillaLauncher.Client/
  ManifestGenerator.cs        — обход папки + хеширование, вынесено из tools/GenerateManifest
                                 (теперь используется и им, и Admin-пайплайном); URL строит
                                 вызывающая сторона через callback — генератор не знает,
                                 куда именно публикуется манифест

VanillaLauncher.Admin/
  GitHubReleaseClient.cs      — тонкий клиент GitHub REST API: создать релиз, загрузить ассет
  GitHubTokenProvider.cs      — токен только из переменной окружения VANILLALAUNCHER_GITHUB_TOKEN,
                                 нигде не хранится в репозитории/appsettings.json
  ReleasePublisher.cs         — генерирует манифест (ManifestGenerator) с плоскими GitHub-
                                 совместимыми URL и заливает его + все файлы в новый релиз
  ServerFileSync.cs           — зеркалирует mods/config из папки сборки в серверную папку
                                 (по хешу, с удалением устаревших файлов)
  PublishPipeline.cs          — оркестрация: бэкап мира -> стоп сервера -> синхронизация ->
                                 публикация манифеста -> старт сервера
```

**Плоские имена ассетов.** GitHub Release assets не поддерживают "/" в имени. `path` в
манифесте остаётся вложенным ("mods/sodium.jar" — так его ищет клиент локально), а имя
ассета на GitHub схлопывается (`ManifestGenerator.FlattenPathForAssetName`: "mods/sodium.jar"
→ "mods_sodium.jar"). `tools/GenerateManifest` получил флаг `--flat` для той же схемы при
ручной генерации; без флага (по умолчанию) сохраняется старое поведение с вложенным путём в
URL — годится для хостинга, который поддерживает папки, но не для реального GitHub Release.

**Каждый релиз самодостаточен.** Ассеты старых релизов не переиспользуются — каждая публикация
перезаливает все файлы заново. Осознанный компромисс: без этого манифест новой версии мог бы
ссылаться на ассет старого релиза, и его удаление (вручную, админом) сломало бы клиентам докачку
задним числом. Заливка ~200+ МБ модов на каждый релиз — цена этой надёжности.

**Пайплайн отказоустойчив только "вперёд".** Если любой шаг падает (не остановился сервер,
не удалось создать релиз, обвалилась сеть на середине аплоада), `PublishPipeline` не пытается
откатить уже сделанное и не запускает сервер обратно — состояние остаётся, где упало, чтобы
админ разобрался, а не чтобы код угадывал, как исправить частично опубликованный релиз.

**Токен.** Только через `VANILLALAUNCHER_GITHUB_TOKEN` — не файл, не appsettings.json, не
что-либо коммитящееся в репозиторий. Нужен PAT с правом `Contents: Read and write` на этот
репозиторий.

`AdminWindow`: поле ввода версии + кнопка "Опубликовать обновление" — подтверждение
(явно предупреждает, что релиз публичный), проверка токена/GitHubOwner/GitHubRepo до
запуска, лог прогресса по всем пяти шагам пайплайна.

**Проверено:** 26 xUnit-тестов (`GitHubReleaseClient`, `ReleasePublisher`, `ServerFileSync`,
`PublishPipeline`) — GitHub API замокан через `FakeHttpMessageHandler`, `PublishPipeline`
гоняется на синтетическом `.bat` (как в Этапе 3) от начала до конца, включая сценарий "сервер
не ответил на stop — пайплайн прерывается до синхронизации файлов".

**Живой прогон выполнен** (после подготовки к релизу) — и вскрыл реальную проблему,
см. следующий раздел.

## Живой прогон публикации и ServerExcludeMods

Первая настоящая публикация (`26.1.2-b1`) отработала пайплайн полностью — бэкап, синхронизация,
загрузка 171 файла в реальный GitHub Release, — но сервер после этого не запустился.

**Что произошло:** до синхронизации на `Server VS` было 122 мода — набор, который никогда не
совпадал 1:1 с клиентской папкой (`ProfileRoot`), это выяснилось только сейчас, потому что
раньше `ServerFileSync` ни разу не запускался на реальных данных. `MirrorAsync` зеркалирует
папку строго (лишнее в серверной папке удаляется) — значит после первого же реального прогона
все 45 модов, которых не было в клиенте, с сервера пропали, а сервер обзавёлся клиентскими
модами клиента, часть которых дедик-серверу не подходит. Один из них, `better_tab`
(`"environment": "*"` в fabric.mod.json — по метаданным универсальный), на деле в
main-entrypoint трогает `net.minecraft.client.gui.screens.Screen` и валит сервер на старте с
`NoClassDefFoundError`. Бэкапа старого набора модов сервера не оказалось — то, что было,
восстановить было неоткуда, договорились считать текущую клиентскую папку источником истины
и чинить дальше от неё (см. CHANGELOG.md).

**Решение — `ServerExcludeMods`** (новое поле `AppConfig`/`appsettings.json`, список имён
файлов): `ServerFileSync.MirrorAsync` принимает необязательный `excludeFileNames` и никогда
не копирует (и удаляет, если уже скопировано) файлы с такими именами на сервер — но именно
на сервер: манифест и клиентская раздача этот список не трогает, клиенты по-прежнему получают
все моды, включая клиентские (`sodium`, `iris` и т.д. — они нужны игрокам, просто не серверу).

Список составлен не на глаз: у каждого мода в `mods/` прочитан `fabric.mod.json` внутри jar
(это обычный zip), поле `environment`. 15 модов оказались явно `"client"`. `better_tab` не
поймать так — он единственный, кого добавили по факту реального краша (метаданные врут,
статический анализ таких случаев не ловит в принципе — это баг конкретного мода). Если в
будущем в модпак добавится новый клиентский мод, который сам не признаётся в этом честно,
единственный надёжный способ узнать — реально попробовать запустить сервер и посмотреть на
трассировку; поэтому это тоже описано в docs/ADMIN_GUIDE.md как процедура, а не только
финальный список.

**Итоговая проверка (всё настоящее, не смоделированное):**
- `Server VS` с 61 модом (77 клиентских минус 16 исключённых) стартовал: `Done (1.236s)! For
  help, type "help"`, без ошибок, штатно остановлен.
- Реальный клиент (свежий запуск, не тот же процесс, что публиковал) сверился с реальным
  манифестом `26.1.2-b1`: `Сборка актуальна`, `171 файл в манифесте`, `Расхождений не
  найдено` — то есть каждый локальный файл побайтово (по SHA-256) совпал с тем, что лежит на
  GitHub.
- 3 новых xUnit-теста на `ServerFileSync` с `excludeFileNames` (не копируется, удаляется, если
  уже было; регистронезависимо).

## Встроенный fallback для appsettings.json

Реальный пользователь скачал `VanillaLauncher.exe` из релиза по инструкции и получил крах:
файл `appsettings.json` в релиз не попал — при публикации `VanillaLauncher.exe` был залит
отдельным ассетом вручную (см. CHANGELOG.md), и про его соседа забыли. `AppConfig.Load()`
до этого момента считал отсутствие внешнего файла настройки фатальной ошибкой
(`FileNotFoundException`) — разумно для разработки, но означает, что судьба каждого релиза
зависит от того, не забыл ли кто-то приложить второй файл.

Исправлено на уровне кода, а не только "не забывать в следующий раз": `appsettings.json`
теперь ещё и `EmbeddedResource` в сборке `VanillaLauncher.Client`
(`LogicalName="VanillaLauncher.Client.appsettings.default.json"`) — тот же файл, что
копируется рядом с exe для локальной разработки, просто ещё и встроен в саму сборку.
`AppConfig.Load()`: если внешнего файла нет — читает встроенный дефолт вместо падения.
`_loadedFromDirectory` в обоих случаях выставляется на папку exe, так что `Save()`
(например, после выбора папки сборки на первом запуске — см. "Подготовка к релизу" ниже)
создаёт полноценный локальный файл вне зависимости от того, откуда был прочитан дефолт.

Проверено реальным сценарием: publish-сборка скопирована в пустую папку без единого
внешнего файла — запускается, работает, не падает. `VanillaLauncher.exe` в уже
опубликованном релизе `26.1.2-b1` заменён на пересобранный (с этим фиксом) через тот же
`GitHubReleaseClient` — старый ассет удалён, новый залит под тем же именем.

## Модули (этап 6 — авторизация Admin-режима)
```
VanillaLauncher.Admin/
  AdminAuthService.cs           — SetPassword/VerifyPassword/HasPassword: соль + PBKDF2-SHA256
                                   (100 000 итераций), сравнение хешей константным по времени

VanillaLauncher.Client.UI/
  SetAdminPasswordWindow.xaml(.cs)  — первый запуск: два поля (пароль + подтверждение)
  AdminLoginWindow.xaml(.cs)        — последующие запуски: одно поле, ошибка не закрывает окно
```
`MainWindow.AdminButton_Click` — гейт: если `AdminAuthService.HasPassword()` == false, показывает
`SetAdminPasswordWindow`; иначе `AdminLoginWindow`. `AdminWindow` создаётся только после
`ShowDialog() == true` — при отмене или неверном пароле не создаётся вообще.

**Файл с хешем.** `admin-auth.json` рядом с exe (там же, где `appsettings.json`) — то есть в
build output, который уже целиком в `.gitignore` (`bin/`); дополнительно сам файл тоже явно
внесён в `.gitignore` — если кто-то поменяет схему копирования файлов в output, хеш пароля всё
равно не закоммитится. Plaintext-пароль нигде не хранится и не логируется, только
соль (16 байт) + PBKDF2-хеш (32 байта, SHA-256, 100 000 итераций).

**"Привязка к ПК" не реализована.** SPEC.md отмечает её как опциональный пункт. Файл с хешем
и так физически лежит на конкретной машине — отдельный крипто-механизм (например, шифрование
через Windows DPAPI, которое бы намертво привязало файл к профилю Windows-пользователя) посчитали
избыточным для приватного сервера на несколько друзей: риск, который он закрывает (кто-то скопировал
`admin-auth.json` на другую машину и брутфорсит пароль оффлайн), не соответствует модели угроз
такого проекта.

**Проверено:** 8 xUnit-тестов на `AdminAuthService` (roundtrip верного/неверного пароля,
персистентность между инстансами, файл никогда не содержит plaintext, разные соли для
одинакового пароля). Сквозная проверка через живое UI Automation: первый запуск → установка
пароля → `AdminWindow` открывается; повторный запуск → неверный пароль отклоняется (диалог
не закрывается), верный пароль пускает.

## Подготовка к релизу (пост-Этап 6)

Три пробела всплыли, когда встал реальный вопрос "можно ли это скачать и запустить":

**1. ProfileRoot — папка конкретного человека, не константа.** Значение в `appsettings.json`,
которое уезжает внутрь exe при сборке, — это папка сборки автора. У любого другого человека,
скачавшего тот же exe, такой папки на диске не будет. Решение: `MainWindow` при старте
проверяет `Directory.Exists(ProfileRoot)`; если папки нет — показывает кнопку "Выбрать папку
сборки" (`Microsoft.Win32.OpenFolderDialog`, нативный, встроен в .NET 8 WPF), выбор
сохраняется через новый `AppConfig.Save()` обратно в **локальный** `appsettings.json` этого
человека. Ничего не коммитится — каждый друг правит только свою копию файла на своей машине.

**2. Необработанные исключения роняли всё приложение.** `AppConfig.Load()` бросает
исключение при отсутствии/поломке файла конфига — без обработки это крашило WPF-приложение
целиком со стектрейсом на экране. Для человека, которому просто прислали exe, это
неприемлемо. Исправлено: `MainWindow.InitializeAsync()` оборачивает загрузку конфига в
try/catch с понятным сообщением в статус-баре; `App.xaml.cs` дополнительно ловит
`DispatcherUnhandledException` глобально — любая необработанная ошибка показывается
`MessageBox`, а не роняет процесс.

**3. Не было ни publish-сборки, ни распространяемого exe.** До этого момента запускали
только через `dotnet run`/`dotnet build` (Debug). Настроена publish-сборка:

```powershell
cd src/VanillaLauncher.Client.UI
dotnet publish -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true
```

`RuntimeIdentifier`/`SelfContained` сознательно **не** зашиты в основной `PropertyGroup`
csproj — если бы они были там безусловно, обычный `dotnet build`/`dotnet run` тоже стал бы
self-contained win-x64 сборкой: другой путь выхода (`bin/Debug/net8.0-windows/win-x64/`
вместо `.../net8.0-windows/`), что сломало бы все скрипты/тесты, завязанные на текущий путь,
и заметно замедлило бы каждую итерацию разработки. `IncludeNativeLibrariesForSelfExtract`/
`EnableCompressionInSingleFile` включены условно (`Condition="'$(PublishSingleFile)' == 'true'"`)
— применяются только во время публикации.

Результат — один exe (`VanillaLauncher.exe`, ~70 МБ, self-contained: другу не нужно ставить
.NET отдельно). Проверено запуском в полностью изолированной папке (без связи с dev-окружением)
с намеренно несуществующим `ProfileRoot` — онбординг (пункт 1) сработал корректно на реальной
publish-сборке, а не только в Debug.

## Модули (этап — настройка лаунчера под модпак)
```
VanillaLauncher.Client/
  PathAutoDetectService.cs   — TryFind(folderName, searchRoots): ищет прямую поддиректорию
                                с совпадающим (регистронезависимо) именем в каждом корне
                                поиска по порядку, не рекурсивно
  appsettings.default.json   — по-настоящему пустой embedded-шаблон движка (см. ниже)

VanillaLauncher.Client.UI/
  SettingsWindow.xaml(.cs)   — экран "Настройки": все Admin-поля из appsettings.json,
                                редактируемые; открывается только изнутри AdminWindow
```
См. docs/TASK_PATH_AUTODETECT.md — постановка задачи и Definition of Done.

**Модель "движок/модпак" — реализовано полностью.** Изначально (в рамках сессии, где
писался код автоопределения путей) репозиторий разделять на два не стали — сначала
сделали то, что относится к коду:
- Embedded-дефолт `AppConfig` (`appsettings.default.json`) по-настоящему пустой — без
  единого значения, специфичного для какого-либо конкретного модпака.
- `.github/workflows/engine-release.yml` — при пуше тега `engine-v*` собирает
  self-contained win-x64 exe и публикует его как GitHub Release этого репозитория,
  отдельно от релизов модпаков.

Позже репозиторий фактически разделили: этот репозиторий (`khanneli-local-launcher-minecraft`)
теперь и есть "движок" — только код, `appsettings.json` здесь тоже по-настоящему пустой (не
только embedded-дефолт). Контент/релизы конкретных модпаков живут в отдельных репозиториях —
пример: [`Pewdoloco/VanillaLauncher-localServer`](https://github.com/Pewdoloco/VanillaLauncher-localServer)
(модпак VanillaScary), из которого в этот репозиторий переехал весь код (`src/`, `tests/`,
`tools/`, этот workflow) и доки о коде/архитектуре; специфика самого модпака VanillaScary
(`docs/SPEC.md`, `docs/ADMIN_GUIDE.md`, `docs/CLIENT_GUIDE.md`, `docs/OVERVIEW.md`) осталась
там. `CHANGELOG.md`/история ниже в этом файле — общая с тем репозиторием до момента
разделения; упоминания VanillaScary/Pewdoloco в исторических разделах — не конфигурация,
а то, на каком реальном деплое движок разрабатывался и проверялся.

**`AppConfig.Load()` больше не бросает исключение на пустых `ManifestUrl`/`ProfileRoot`.**
Раньше отсутствие `ManifestUrl` в конфиге считалось ошибкой чтения файла
(`InvalidOperationException`). Но у "движка" без адаптации под модпак пустой конфиг —
это ожидаемое, валидное состояние, а не повреждённый файл. Вместо исключения — новое
свойство `AppConfig.IsConfigured` (`ManifestUrl`/`GitHubOwner`/`GitHubRepo` все заданы).
`ProfileRoot` в `IsConfigured` сознательно не участвует: это per-machine значение (у
каждого игрока — свой путь), которое легитимно пустое даже на полностью адаптированном
под модпак движке, пока конкретный игрок ещё не прошёл онбординг. `MainWindow` теперь
различает эти два случая явно: `!IsConfigured` → "лаунчер не настроен под модпак,
обратись к администратору" (кнопки проверки обновлений недоступны, `SelectProfileRootButton`
не показывается — выбирать путь смысла нет, если сам модпак не настроен); `IsConfigured`,
но `ProfileRoot` не существует на диске → обычный, ожидаемый онбординг конкретного игрока
(сначала автопоиск, потом ручной выбор).

**Автопоиск — `PathAutoDetectService.TryFind`.** Не рекурсивный, ищет только прямых потомков
переданных корней по точному (регистронезависимому) совпадению имени папки. Рекурсивный
обход всего диска сознательно не делали — непредсказуемо долго на больших дисках и находит
случайные совпадения имени в неожиданных местах.
- **Клиент:** корни поиска — `AppConfig.ClientSearchRoots` (список путей из конфига,
  например `D:\Games\curseforge\Instances`), имя — `ClientFolderName`. Ничего не
  подставляется автоматически, если оба поля пустые (типично для свежего, ещё не
  адаптированного под конкретную CurseForge-инсталляцию движка) — тогда сразу фоллбек на
  ручной выбор, как и раньше.
- **Сервер:** корни поиска — всегда папка рядом с `.exe` (`AppContext.BaseDirectory`, это
  и есть сценарий "создаётся рядом с лаунчером" из задачи) плюс опциональные
  `AppConfig.ServerSearchRoots` для нестандартных раскладок. Сценарий "инициализируется при
  старте лаунчера" отдельно не реализован как автоматическое создание папки — вместо этого
  ручной фоллбек использует `OpenFolderDialog`, который сам умеет создавать новую папку
  (кнопка "Создать папку" в диалоге), так что результат тот же без отдельного кода.
- Обе автодетект-точки (`MainWindow` для клиента, `AdminWindow` для сервера) при успехе сразу
  вызывают `AppConfig.Save()` — совпадение перепроверяется на каждом запуске (не кешируется
  отдельно), но раз найденный путь пишется в `appsettings.json`, так что при следующем
  запуске уже сам факт `Directory.Exists(ProfileRoot/ServerDirectory)` "срезает" повторный
  автопоиск, если ничего на диске не поменялось.

**Экран "Настройки" — `SettingsWindow`.** Не гейтится паролём отдельно — открывается только
кнопкой внутри уже открытого `AdminWindow` (`SettingsButton_Click`), то есть проход через
тот же `AdminAuthService`-гейт, что и весь Admin-режим, уже произошёл к этому моменту.
`MainWindow` про `SettingsWindow` вообще не знает — у обычного пользователя нет пути
достижения этого окна ни при каких обстоятельствах. Поля-списки (`ClientSearchRoots`,
`ServerSearchRoots`, `IncludeFolders`, `ServerExcludeMods`) редактируются как многострочный
текст, одна запись на строку — самый простой ввод, не требующий кастомного list-редактора
для полей, которые правятся раз в сто запусков. Кнопки "Найти автоматически" в этом окне
гоняют `PathAutoDetectService` по ещё не сохранённым значениям полей формы — позволяет
проверить имя/корни поиска до `Save()`, не перезапуская лаунчер.

## Модули (этап — env-переменные в путях, аудит модов из папки, встроенная инструкция)

Три доработки поверх этапа "настройка лаунчера под модпак" (запрошены после того, как
модель "движок/модпак" уже разошлась по отдельным репозиториям).

**Переменные окружения в `ClientSearchRoots`/`ServerSearchRoots`.**
`PathAutoDetectService.TryFind` перед проверкой каждого корня прогоняет его через
`Environment.ExpandEnvironmentVariables` (раскрывает `%USERPROFILE%`, `%LOCALAPPDATA%` и
любые другие Windows-переменные). Причина: `ClientSearchRoots` раньше приходилось задавать
абсолютным путём одного конкретного человека (`D:\Games\curseforge\Instances`) — не
работает у друзей с другим именем пользователя или диском. `%VAR%`, которого нет в
окружении, `ExpandEnvironmentVariables` не бросает исключение, а оставляет строку как
есть — она просто не найдётся на диске и тихо пропустится (как отсутствующий корень),
автопоиск перейдёт к следующему корню/фоллбеку. Найденный `TryFind` путь — уже
раскрытый конкретный путь этой машины, сохраняется в конфиг как обычно.

**Выбор модов из папки для `ServerExcludeMods` — `SelectExcludedModsWindow`.**
Раньше список редактировался только как текст (имя файла на строку) — нужно было точно
знать/переписывать имена jar вручную. Новое окно (открывается кнопкой в `SettingsWindow`,
по умолчанию подставляет `{ProfileRoot}\mods`) сканирует `*.jar` в выбранной папке
(`SearchOption.TopDirectoryOnly` — плоский список, как реальная папка модов) и показывает
чекбоксы, предварительно отмеченные по текущему `ServerExcludeMods`. Результат **не**
затирает список целиком: `SettingsWindow.PickExcludedMods_Click` мёржит отмеченные файлы
из просканированной папки с записями, которых в этой папке не было (например, введёнными
вручную для другого набора модов) — так пикер нельзя случайно использовать, чтобы стереть
не относящиеся к текущей папке исключения. Автоматизации самого решения "клиентский мод
или нет" здесь по-прежнему нет (см. "Что сознательно отложено") — это только замена
ручного набора текста на чекбоксы по реальному списку файлов.

**Встроенная инструкция — `GuideWindow` + `AppConfig.ClientGuide*`/`AdminGuide*`.**
Четыре новых строковых поля конфига (`ClientGuideShort`, `ClientGuideFull`,
`AdminGuideShort`, `AdminGuideFull`) с непустыми дефолтами из `DefaultGuides`
(`VanillaLauncher.Client`) — в отличие от путей/`ManifestUrl`, это осознанно не пустой
дефолт: сам по себе текст инструкции не содержит ничего специфичного для конкретного
модпака (описывает кнопки/поведение движка), поэтому он такой же generic-дефолт, как
`ServerBatFileName = "start.bat"` или `IncludeFolders = ["mods", "config"]` выше. Админ
переопределяет текст под свой модпак через `SettingsWindow` (четыре новых многострочных
поля внизу экрана), обычным `Save()`.

`GuideWindow` открывается кнопкой "Инструкция" в `MainWindow` (клиентский текст) и в
`AdminWindow` (админский) через `Show()`, не `ShowDialog()`, и без установки `Owner` в
смысле модальности (Owner проставлен только ради z-order/сворачивания вместе с
родительским окном) — специально, чтобы инструкцию можно было держать открытой и
одновременно продолжать пользоваться остальным лаунчером (например, сверяться с шагами
публикации, пока сама публикация идёт). Переключение "Краткое"/"Полное" — два `RadioButton`
на немедленную подмену текста в одном read-only `TextBox`; без прогонки текста markdown/HTML
разметки во что-то богаче, обычный текст (в кратком/полном формате) для этой задачи
достаточен.

## Что сознательно отложено
- Переиспользование ассетов между релизами (см. выше, почему не делали)
- Привязка Admin-пароля к конкретному ПК (см. выше, почему не делали)
- Автообновление самого лаунчера (публикация модпака не трогает exe — см. docs/ADMIN_GUIDE.md)
- Цифровая подпись exe (SmartScreen может ругаться на первый запуск — см. docs/CLIENT_GUIDE.md)
- Частичная автоматизация аудита `ServerExcludeMods` ("кнопка проверить моды" из открытых
  вопросов задачи) — осталась ручной процедурой; в этом репозитории (движок без конкретного
  модпака) её и не может быть — см. `docs/ADMIN_GUIDE.md` в репозитории модпака
