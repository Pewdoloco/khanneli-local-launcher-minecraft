# Задача — настройка лаунчера под модпак: пути, экран "Настройки", модель распространения

**Статус: реализовано полностью**, включая фактическое разделение репозиториев — этот
репозиторий (`khanneli-local-launcher-minecraft`) и есть "движок", контент модпака
VanillaScary остался в `Pewdoloco/VanillaLauncher-localServer`. См. `docs/ARCHITECTURE.md`,
раздел "Модули (этап — настройка лаунчера под модпак)" для деталей реализации и решённых
открытых вопросов.

Изначально задача была узкой ("автоопределение путей клиент/сервер") — по итогам
обсуждения расширена: тот же UI-экран должен закрывать не только пути, но и вообще все
настройки, нужные, чтобы адаптировать один и тот же лаунчер под произвольный модпак
(другой загрузчик, другой репозиторий) без правки кода. История обсуждения ниже сведена
в решения — переоткрывать их не нужно.

---

## Модель распространения (решено)

Идея: один "движок" — много "модпак-деплойментов", без привязки кода к конкретной
сборке.

- **Репозиторий "движок"** — только исходный код лаунчера + по-настоящему пустой
  дефолтный конфиг (никаких значений вроде `Pewdoloco`/`VanillaScary`, зашитых сейчас в
  `EmbeddedResource`). Публикует **свои** GitHub Releases с тегами вида `engine-vX.Y.Z`
  (отдельная нумерация, не пересекается с версиями модпаков) — готовый "чистый" exe можно
  скачать напрямую, не собирая локально через `dotnet publish` каждый раз, когда
  адаптируешь под новый модпак.
- **Репозитории "модпак"** (например, будущий Khazad) — только хостят релизы: `manifest.json`
  + файлы модов + (опционально) уже настроенный под этот модпак exe как ассет. Кода в них
  нет вообще — это чисто содержательные репозитории, ровно как `VanillaLauncher-localServer`
  используется сейчас (только там код и релизы смешаны в одном репо).
- **Токен.** Один `VANILLALAUNCHER_GITHUB_TOKEN`, меняется вручную при переключении между
  тем, какой модпак сейчас администрируешь с этой машины. Осознанно не делаем настраиваемое
  имя переменной/хранилище нескольких токенов — усложнение не оправдано при том, что
  переключение между проектами — редкое действие, не десятки раз в день.

## Admin: отдельный экран "Настройки" (решено)

Не часть текущего `AdminWindow` — отдельное окно, доступное только после входа по паролю
(тот же гейт, что и у `AdminWindow`). Причина разделения: "оперативные" действия
(старт/стоп/публикация) в `AdminWindow` используются часто, "конфигурационные" — редко,
только при первичной адаптации под новый модпак или смене инфраструктуры. Смешивать их в
одном окне не нужно.

### Поля экрана "Настройки" и видимость по ролям

| Поле | Admin | Пользователь (Client) |
|---|---|---|
| Путь клиентской сборки (`ProfileRoot`) | видит, редактирует | видит, редактирует (как сейчас, через онбординг) |
| Путь серверной директории (`ServerDirectory`) | видит, редактирует | не видит |
| `ServerBatFileName` | видит, редактирует | не видит |
| `ManifestUrl` / `GitHubOwner` / `GitHubRepo` | видит, редактирует | не видит |
| `IncludeFolders` | видит, редактирует | не видит |
| `ServerExcludeMods` | видит, редактирует (список) | не видит |
| `MaxBackupsToKeep` | видит, редактирует | не видит |

Обычный пользователь никогда не должен попадать в этот экран целиком — у него по-прежнему
только точечный онбординг пути клиента (см. ниже), без доступа к остальным полям.

## Логика: клиентская директория

1. Пользователь создаёт пустую сборку в CurseForge (или любом другом инструменте).
2. Лаунчер ищет директорию сборки по совпадению имени с ожидаемым названием (пример:
   `VanillaScary`).
3. Если совпадение найдено — путь подставляется автоматически.
4. Если имя другое или совпадение не найдено — пользователь указывает путь вручную.
5. Найденная/указанная директория используется как клиентская папка игры.

## Логика: серверная директория

Два возможных сценария происхождения папки:
- (a) создаётся рядом с `.exe` лаунчера;
- (b) инициализируется при старте лаунчера.

Определение пути:
1. Поиск по совпадению имени с ожидаемым названием (пример: `VanillaScary-Server`).
2. Если совпадение найдено — путь подставляется автоматически.
3. Если имя другое / совпадение не найдено — пользователь указывает путь вручную.

**Архитектурное требование (как и раньше):** ожидаемые имена директорий (клиент и
сервер) — это настройки в конфиге, редактируемые в экране "Настройки", а не
захардкоженные строки в коде.

## Definition of Done

- [x] Admin видит оба пути (клиент + сервер) в экране "Настройки" с возможностью
      редактирования каждого (`SettingsWindow`)
- [x] Пользователь видит только путь клиента (через существующий онбординг, не через
      экран "Настройки") — `MainWindow` не создаёт `SettingsWindow` ни при каких условиях
- [x] Автоопределение клиентского пути по конфигурируемому имени сборки работает; при
      неудаче — фоллбек на ручной выбор папки (`PathAutoDetectService` + `MainWindow`)
- [x] Автоопределение серверного пути по конфигурируемому имени работает (папка рядом с
      .exe **или** созданная при старте); при неудаче — фоллбек на ручной выбор
      (`PathAutoDetectService` + `AdminWindow`; "создана при старте" покрыто через
      `OpenFolderDialog`, который сам умеет создавать новую папку — см. ARCHITECTURE.md)
- [x] Имена директорий (клиент/сервер) не захардкожены — хранятся в настройках/конфиге
      (`AppConfig.ClientFolderName`/`ServerFolderName`)
- [x] Экран "Настройки" в Admin: `ManifestUrl`/`GitHubOwner`/`GitHubRepo`,
      `ServerBatFileName`, `IncludeFolders`, `ServerExcludeMods`, `MaxBackupsToKeep` —
      все редактируемые, доступны только после входа по паролю (`SettingsWindow`
      открывается только изнутри уже авторизованного `AdminWindow`)
- [x] Дефолтный embedded-конфиг "движка" не содержит значений, специфичных для
      VanillaScary/Pewdoloco (генерируется по-настоящему пустым для новой,
      неадаптированной сборки) — `appsettings.default.json`
- [x] Репозиторий "движка" публикует собственные релизы exe (`engine-vX.Y.Z`),
      отдельно от релизов конкретных модпаков — `.github/workflows/engine-release.yml`.
      Код и весь этот репозиторий (`khanneli-local-launcher-minecraft`) теперь и есть
      "движок" — содержимое модпака VanillaScary осталось в
      `Pewdoloco/VanillaLauncher-localServer`, кода там больше нет

## Текущее состояние кода (на момент постановки задачи, после релиза 26.1.2-b1)

Чтобы не переоткрывать уже известное:

- **`AppConfig`** (`src/VanillaLauncher.Client/AppConfig.cs`) хранит `ProfileRoot`
  (клиент) и `ServerDirectory` (сервер, nullable) как обычные строки в
  `appsettings.json`. Никакого автоопределения по имени нет — оба пути захардкожены в
  конфиге при сборке под конкретного пользователя (сейчас
  `D:\Modding\Servers\Client\VanillaScary` и `D:\Modding\Servers\Server VS`).
- **`AppConfig.Save()`** уже умеет перезаписывать `appsettings.json` — пригодится и для
  путей, и для остальных полей экрана "Настройки".
- **`AppConfig`** реализует fallback на `EmbeddedResource`, если внешнего
  `appsettings.json` нет вообще (см. `CHANGELOG.md`, "Фикс: appsettings.json не попал в
  релиз"). Для "движка" этот embedded-дефолт должен стать по-настоящему пустым шаблоном
  (сейчас в нём — конкретные значения VanillaScary/Pewdoloco/16 исключённых модов).
- **Онбординг `ProfileRoot` уже частично есть**, но по другому сценарию: в
  `MainWindow.InitializeAsync()` (`src/VanillaLauncher.Client.UI/MainWindow.xaml.cs`) —
  если `Directory.Exists(ProfileRoot)` == false, показывается кнопка "Выбрать папку
  сборки" → `Microsoft.Win32.OpenFolderDialog` → `Save()`. Это **не автоопределение по
  имени** — просто проверка "существует ли путь из конфига". Новая задача расширяет это:
  сначала пробовать найти по имени, только потом — ручной выбор.
- **Для `ServerDirectory` аналогичного онбординга нет вообще** — Admin-режим просто
  использует значение из конфига как есть.
- **UI путей/настроек нет нигде** — ни в `MainWindow`, ни в `AdminWindow` пути и
  остальные Admin-поля не отображаются как редактируемые, только используются "за
  кадром". Нужен новый экран (см. "Admin: отдельный экран Настройки" выше).
- **`ServerExcludeMods`** заполняется сейчас полностью вручную (см.
  `docs/ADMIN_GUIDE.md`, процедура проверки мода на клиентскость) — экран "Настройки"
  должен дать редактировать этот список, но сам аудит модов автоматизировать не
  пытается (см. открытые вопросы ниже, почему).

## Открытые технические вопросы (на усмотрение реализующей сессии)

- Формат хранения "ожидаемых имён" для автопоиска (новые поля `AppConfig`, например
  `ClientFolderNamePattern`/`ServerFolderNamePattern`?).
- Где именно искать директории при автопоиске (папка CurseForge instances по умолчанию?
  рядом с .exe? заданная в конфиге корневая папка поиска?).
- Поведение при повторном запуске: перепроверять автоопределение каждый раз или
  использовать сохранённый путь, пока пользователь не сбросит вручную.
- **Как отличить "путь не определился" от "лаунчер вообще не адаптирован под модпак"?**
  Сейчас единственный сигнал — `Directory.Exists(ProfileRoot) == false`. Если конфиг
  полностью пустой (свежий "движок", ещё никто не настраивал `ManifestUrl`/`GitHubOwner`
  и т.д.), это качественно другая ситуация, чем "у конкретного игрока просто другой
  путь" — возможно, стоит показывать разный UI/сообщение для этих двух случаев, чтобы не
  запутать человека, который случайно запустил неадаптированный exe.
- **Стоит ли частично автоматизировать аудит `ServerExcludeMods`?** Сейчас это ручной
  процесс (открыть каждый jar, прочитать `fabric.mod.json`, проверить `environment`).
  Можно добавить в экран "Настройки" кнопку "Проверить моды" — просканировать `mods/` и
  предложить список кандидатов с `"environment": "client"`, оставив на ручную проверку
  только случаи вроде `better_tab` (метаданные врут, ловится только реальным запуском
  сервера). Не решено, входит ли это в объём данной задачи или откладывается отдельно.
