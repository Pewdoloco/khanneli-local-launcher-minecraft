Пользователь прочитал текущие `ClientGuideFull`/`AdminGuideFull` (встроенное окно

### 1. Убрать Markdown-разметку из текста инструкций

`GuideWindow.GuideTextBox` — обычный `TextBox` (`IsReadOnly="True"`), НЕ рендерит Markdown.
Значит `# Заголовок`, `**жирный текст**` и т.п. в `DefaultGuides.cs` (`ClientShort`,
`ClientFull`, `AdminShort`, `AdminFull`) показываются пользователю буквально, с решётками
и звёздочками — выглядит как "экранизация readme", а не как реальная инструкция. Нужно
пройтись по всем четырём константам и убрать markdown-синтаксис, заменить на простое
форматирование, которое нормально смотрится в plain-text (пустые строки между разделами,
нумерованные списки, `—`/дефисы вместо `**bold**`, заголовки просто как отдельная строка
без `#`, возможно КАПСОМ или с двоеточием). Заодно перечитать оба текста целиком — часть
формулировок могла устареть за последние ~7 версий (`engine-v1.3.0`…`v1.7.0`), не только
markdown-синтаксис.

### 2. Добавить новый раздел "Руководство по использованию" в окно "Инструкция"

Пользователь написал два пошаговых гайда (Клиент + Админ) в чате, я их поправил под
реальный интерфейс и опубликовал как артефакт:
https://claude.ai/code/artifact/98bb10b5-adec-4ac6-b7b6-32c2b354d52b — это и есть контент,
который нужно добавить (с правками ниже, п.3). Пользователь хочет видеть их в лаунчере как
**отдельный пункт** в "Инструкции", не смешивая с существующими Краткое/Полное.

Технически: `GuideWindow` сейчас — Role (Пользователь/Администратор) × Length
(Краткое/Полное) = 4 текста (`ClientGuideShort/Full`, `AdminGuideShort/Full`). Естественный
путь — добавить третий вариант в группу длины (`GuideLength` RadioGroup в
`GuideWindow.xaml`): "Краткое" / "Полное" / третий вариант (назвать, например, "Пошаговое
руководство" — сверить с пользователем формулировку, если есть время спросить, иначе
использовать его же "Руководство по использованию"). Понадобится:
- 2 новых поля в `AppConfig.cs` (например `ClientGuideManual`/`AdminGuideManual` — сверить
  с существующей конвенцией именования `ClientGuideShort/Full`).
- 2 новых константы в `DefaultGuides.cs` с текстом из артефакта (после правок п.3).
- `GuideWindow.xaml` — третий `RadioButton` в группу `GuideLength`.
- `GuideWindow.xaml.cs` — `UpdateText()` сейчас `switch` на `(bool isAdmin, bool isFull)`,
  нужно расширить до трёх состояний длины (enum или дополнительный bool/switch-паттерн).
- `SettingsWindow.xaml`/`.xaml.cs` — добавить ещё пару editable-полей (по образцу уже
  существующих 4 полей руководств: `TextBox` + кнопка "Редактировать"/"Готово",
  `ToggleGuideEdit_Click` уже параметризован через `Tag`, просто добавить кейсы
  `"ClientManual"`/`"AdminManual"` в `GetGuideField`).

### После правок

- Пересобрать (`dotnet build` + `dotnet test` оба проекта — 119 тестов, ни один не должен
  сломаться, изменения только в UI-тексте/разметке `GuideWindow`, логики не касаются).

  `EngineVersion.Current` (`src/VanillaLauncher.Client/EngineVersion.cs`) с `1.7.0` на

  свежая всегда сверху), запушить тег `engine-v1.8.0`, дождаться `conclusion: success` в
  GitHub Actions (`gh`/`curl -s https://api.github.com/repos/Pewdoloco/
  khanneli-local-launcher-minecraft/actions/runs?per_page=1`), проверить, что exe-ассет
  реально появился в Releases.

## Как проверялась работа в прошлых сессиях (нет своего CLI-тула, только это)

Для живого прогона (сборка + запуск + клики) использовался PowerShell +
`System.Windows.Automation` — рабочий хелпер-скрипт лежит НЕ в репозитории (временный,
per-session), нужно будет пересобрать заново по описанию:
`Add-Type -AssemblyName UIAutomationClient, UIAutomationTypes` + функции
`Find-ByName`/`Invoke-Element`/`Type-IntoElement` (через `SendKeys`, `PasswordBox`
блокирует `ValuePattern`) + fallback на реальный клик мышью (`SetCursorPos`+`mouse_event`)
для нативных `MessageBox`-кнопок, которые иногда экспонируются как `Pane` без
`InvokePattern`, а не `Button`. GitHub Actions runs проверяются напрямую через
`api.github.com/repos/.../actions/runs` (без токена, публичный репозиторий).

## Важно знать перед продолжением

- **`test-pack` (github.com/Pewdoloco/test-pack) — удалён/приватный.** Если нужен живой
  прогон публикации — создать новый тестовый репозиторий (публичный, с README при
  создании — иначе 422 "Repository is empty" на первой публикации).
- **Файловая структура на машине пользователя может измениться между сессиями без
  предупреждения** — не полагаться на пути из истории чата не глядя, перепроверять `ls`/
  `Get-ChildItem` в начале работы.
- Тестовые папки прошлой сессии (`D:\Modding\Servers\ClientSim`,
  `D:\Modding\Servers\PlayerSim`) пользователю предложено удалить как чисто тестовые —
  если они всё ещё существуют на диске, можно переиспользовать или пересоздать заново,
  не критично.
