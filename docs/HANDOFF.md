# HANDOFF — состояние на 2026-07-21, как продолжить в новой сессии

Этот файл — снимок состояния проекта для новой сессии Claude Code (или для себя через
пару месяцев). Если начинаешь новую сессию — открой этот файл, `docs/ARCHITECTURE.md`
и `docs/TASKS.md`, дальше не нужно пересказывать весь путь заново.

## Что это вообще такое (30 секунд)

Два репозитория:
- **[khanneli-local-launcher-minecraft](https://github.com/Pewdoloco/khanneli-local-launcher-minecraft)**
  (этот репозиторий) — "движок": весь код лаунчера, ни к какому конкретному модпаку не
  привязан. Публикует свои релизы `engine-vX.Y.Z` — самодостаточный exe.
- **[VanillaLauncher-localServer](https://github.com/Pewdoloco/VanillaLauncher-localServer)**
  — "модпак": контент (моды, `manifest.json`) для конкретного сервера VanillaScary,
  публикуется через Admin-режим exe, собранного из этого репозитория и настроенного под
  VanillaScary через экран "Настройки". Кода там нет вообще.

Один exe, два режима: **Client** (игрок — докачка модов, самообновление лаунчера) и
**Admin** (владелец сервера — публикация, старт/стоп сервера, конфигурация).

## Текущая версия

`EngineVersion.Current` = **1.2.0** (`src/VanillaLauncher.Client/EngineVersion.cs`),
последний опубликованный тег — `engine-v1.2.0`. При следующем релизе: бампнуть эту
константу **и** запушить тег `engine-vX.Y.Z` с тем же числом — CI
(`.github/workflows/engine-release.yml`) проверяет совпадение и падает при расхождении.

## Полностью реализовано и проверено живым прогоном

- Автоопределение путей клиента/сервера по имени папки, с поддержкой `%VAR%`
  (`PathAutoDetectService`), фоллбек на ручной выбор.
- Экран "Настройки" (`SettingsWindow`, Admin-only): все поля `AppConfig` редактируемые,
  включая автопоиск, выбор модов из папки с поиском/пагинацией, руководства с
  защитой-от-случайного-ввода.
- Встроенное окно "Инструкция" (`GuideWindow`): роль (Пользователь/Администратор) +
  длина (Краткое/Полное) в одном немодальном окне, устойчивом к чужой модальности
  (WM_ENABLE hook).
- Самообновление exe (`EngineSelfUpdater`) — проверка/скачивание/подмена файла/рестарт,
  проверено на реальном скачивании с GitHub.
- Публикация обновлений модпака одной кнопкой, старт/стоп сервера, бэкапы мира с
  ротацией, авторизация Admin-режима.

Полная историческая хронология решений — `CHANGELOG.md` (сверху вниз, новое — выше) и
`docs/ARCHITECTURE.md` (по разделам "Модули (этап — …)").

## Открытые вопросы / что осталось (в порядке убывания приоритета)

1. **Фактическая проверка авто-найденного разрешения UIA-квирков в тестах не относится к
   продукту** — при живом тестировании изредка `TogglePattern`/`InvokePattern` не находили
   элемент с первого раза через `FindFirst(NameProperty=...)` (возможно, WPF-виртуализация
   `ListBox`). Это issue тестовой автоматизации, не приложения — но если пишешь ещё
   UI-тесты руками через UI Automation, учти и используй запасной путь через реальные
   координаты мыши (`SetCursorPos`+`mouse_event`), как в этой сессии.
2. **Проверка целостности скачанного exe при самообновлении** (SHA-256/подпись) не
   реализована — осознанный компромисс (см. ARCHITECTURE.md, "Что сознательно
   отложено"). Если станет важно — GitHub Release Assets API отдаёт поле `digest`
   (`sha256:...`) для новых загрузок, можно свериться после скачивания в
   `EngineSelfUpdater.PrepareAndLaunchUpdateAsync`.
3. **Аудит `ServerExcludeMods`** — до сих пор ручная процедура (открыть каждый jar,
   прочитать `fabric.mod.json`, поле `environment`). Кнопка "Проверить моды", которая бы
   сама сканировала `environment` и подсказывала кандидатов, не реализована — см.
   `docs/TASK_PATH_AUTODETECT.md` в этом репозитории, "Открытые технические вопросы", это
   решение принято сознательно не делать сразу (некоторые моды типа `better_tab` лгут в
   метаданных, ловятся только реальным запуском сервера — автоматизация была бы неполной).
4. **Цифровая подпись exe** не сделана — Windows SmartScreen может ругаться на первый
   запуск. См. `docs/CLIENT_GUIDE.md` в репозитории модпака, там уже есть инструкция
   "Подробнее → Выполнить в любом случае" для игроков.
5. **Привязка Admin-пароля к конкретному ПК** не сделана — осознанно отложено ещё в
   Этапе 6 (модель угроз для приватного сервера на несколько друзей не требует).

## Прямые ответы на вопросы (были заданы в чате, дублирую сюда, чтобы не потерялись)

**GitHubOwner/GitHubRepo vs EngineGitHubOwner/EngineGitHubRepo — разница?**
Формально одна и та же пара полей (Owner = юзернейм/организация на GitHub, Repo = имя
репозитория внутри), но указывают на РАЗНЫЕ репозитории: `GitHubOwner`/`GitHubRepo` — куда
публикуется контент модпака (например, `Pewdoloco`/`VanillaLauncher-localServer`);
`EngineGitHubOwner`/`EngineGitHubRepo` — откуда качать новую версию самого exe (обычно
`Pewdoloco`/`khanneli-local-launcher-minecraft`, то есть этот репозиторий). Оба поля теперь
подписаны в `SettingsWindow` явно ("Репозиторий МОДПАКА" / "Репозиторий ДВИЖКА").

**Манифест (manifest.json) — кто создаёт, откуда брать ссылку?**
Обычно создаётся автоматически: кнопка "Опубликовать обновление" в Admin-режиме
(`PublishPipeline`) сама генерирует `manifest.json` из содержимого `ProfileRoot` и
загружает его вместе с модами в новый GitHub Release. `ManifestUrl` в "Настройках" —
это просто `https://github.com/{GitHubOwner}/{GitHubRepo}/releases/latest/download/manifest.json`
— вычисляется один раз вручную по этой формуле и дальше не меняется: GitHub сам
перенаправляет `/latest/` на самый свежий релиз при каждом запросе. Ручная генерация
(`tools/GenerateManifest` в этом репозитории) — редкий случай не-GitHub раздачи,
задокументирована в `docs/ADMIN_GUIDE.md` репозитория модпака.

**Бэкапы мира — куда сохраняются, создаётся ли папка сама, максимум 5?**
`{ServerDirectory}\backups\` — папка создаётся автоматически при первом бэкапе
(`Directory.CreateDirectory` в `WorldBackupService.BackupWorld`, `src/VanillaLauncher.Admin/WorldBackupService.cs`).
`MaxBackupsToKeep` = 5 — это только ДЕФОЛТНОЕ значение, не жёсткий максимум; можно
поменять на любое целое ≥ 1 через "Настройки" (0 теперь корректно отклоняется при
сохранении — см. CHANGELOG, "Фикс валидации").

## Как продолжить в новой сессии

Открой `docs/ARCHITECTURE.md`, `docs/TASKS.md` и этот файл, затем:

> Прочитай docs/HANDOFF.md, docs/ARCHITECTURE.md и docs/TASKS.md в репозитории
> khanneli-local-launcher-minecraft. Дальше: [конкретная задача].

Для живого прогона (сборка + запуск + проверка через UI Automation + PrintWindow-скриншоты)
в этой сессии наработан рабочий PowerShell-скрипт захвата окна напрямую по хендлу
(`PrintWindow` + ручной PNG-энкодер, без System.Drawing — в этом окружении PowerShell не
резолвил зависимости GDI+). Если понадобится повторить — попроси предыдущую сессию
восстановить этот скрипт по описанию в `CHANGELOG.md`/истории разговора, он не сохранён
в репозитории (это была временная тулза для тестирования, не часть продукта).

**Важно:** локальный конфиг для тестов (`appsettings.json` в build output) — тестовый
и не коммитится, никогда не путай его с `src/VanillaLauncher.Client/appsettings.json`
(loose dev-копия в репозитории движка, тоже пустая) или с реальным конфигом на боевой
машине администратора (не в git вообще, настраивается через "Настройки" один раз при
адаптации exe под VanillaScary).
