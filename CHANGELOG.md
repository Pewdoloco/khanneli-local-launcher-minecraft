# Changelog

## Самообновление exe движка

Раньше публикация модпака никогда не трогала сам `VanillaLauncher.exe` — новую версию
лаунчера можно было получить только вручную от админа. Теперь exe умеет проверять и
накатывать обновления сам (полная сводка — docs/ARCHITECTURE.md, раздел "Самообновление exe
движка").

- **`EngineVersion.Current`** — константа с версией этой сборки движка; CI
  (`.github/workflows/engine-release.yml`) сверяет её с тегом `engine-vX.Y.Z` при публикации
  и проваливает релиз при расхождении.
- **`EngineSelfUpdater`** — `CheckForUpdateAsync` (GitHub Releases API, сравнение версий через
  `System.Version`, без токена) и `PrepareAndLaunchUpdateAsync` (скачивает новый exe, готовит
  и запускает `.bat`, который дожидается завершения текущего процесса по PID, подменяет файл
  и перезапускает).
- **`AppConfig.EngineGitHubOwner`/`EngineGitHubRepo`** — отдельно от `GitHubOwner`/`GitHubRepo`
  (те про репозиторий модпака с контентом) — репозиторий, откуда качать новую версию exe.
- **`MainWindow`**: кнопки "Проверить обновление лаунчера"/"Обновить лаунчер" — по аналогии
  с существующей парой для контента модпака. Осознанно в клиентском окне, а не только в
  Admin — целевая аудитория в первую очередь игроки со старыми копиями exe.
- 7 новых тестов на `EngineSelfUpdater.CheckForUpdateAsync` (новее/такая же/старее версия,
  тег без префикса `engine-v`, релиз без нужного ассета, HTTP-ошибка, User-Agent заголовок) +
  2 на `AppConfig` (дефолты и сохранение `EngineGitHubOwner`/`Repo`) — 82 теста всего.
- Сознательно не сделано: проверка целостности скачанного exe (SHA-256/подпись) — TLS
  обеспечивает подлинность канала загрузки, отдельной сверки хеша нет.

## Переменные окружения в путях поиска, выбор модов из папки, встроенная инструкция

Три доработки экрана "Настройки" по итогам живого использования уже разделённого
движка (полная сводка решений — в docs/ARCHITECTURE.md, соответствующий раздел).

- **`%VAR%` в `ClientSearchRoots`/`ServerSearchRoots`** — `PathAutoDetectService.TryFind`
  раскрывает переменные окружения (`%USERPROFILE%` и т.д.) перед проверкой каждого корня.
  Раньше корень поиска приходилось задавать абсолютным путём одного конкретного человека —
  не работало у друзей с другим именем пользователя/диском.
- **`SelectExcludedModsWindow`** — кнопка в "Настройках" сканирует `*.jar` в папке (по
  умолчанию `{ProfileRoot}\mods`) и даёт отметить `ServerExcludeMods` чекбоксами вместо
  ручного набора имён файлов текстом. Мёржит с уже существующими записями, не относящимися
  к просканированной папке, — ничего не стирает случайно.
- **`GuideWindow`** — немодальное окно "Инструкция" (`Show()`, не `ShowDialog()`), доступно
  из `MainWindow` (клиентский текст) и `AdminWindow` (админский), с переключением
  Краткое/Полное. Текст — новые поля `AppConfig` (`ClientGuideShort/Full`,
  `AdminGuideShort/Full`) с непустыми generic-дефолтами (`DefaultGuides`), редактируется
  под конкретный модпак через "Настройки".
- Ручная генерация `manifest.json` (`tools/GenerateManifest`) задокументирована в
  `docs/ADMIN_GUIDE.md` репозитория модпака — раньше существовала только в коде.
- Новые тесты в `PathAutoDetectServiceTests` (раскрытие `%VAR%`, неизвестная переменная не
  ломает автопоиск) и `AppConfigTests` (дефолты/сохранение текста инструкции).

## Фактическое разделение репозиториев: движок отдельно от модпака

Продолжение предыдущего пункта (модель "движок/модпак" была реализована на уровне кода, но
физически всё ещё лежала в одном репозитории, `Pewdoloco/VanillaLauncher-localServer`).
Теперь разнесено по-настоящему:

- **`Pewdoloco/khanneli-local-launcher-minecraft`** (этот репозиторий) — весь код
  (`src/`, `tests/`, `tools/`), CI-публикация `engine-vX.Y.Z`
  (`.github/workflows/engine-release.yml`), доки о коде/архитектуре (`docs/ARCHITECTURE.md`,
  `docs/TASKS.md`, `docs/TASK_PATH_AUTODETECT.md`). `appsettings.json` здесь тоже
  по-настоящему пустой (не только embedded `appsettings.default.json`, как было раньше,
  пока репозитории не были разделены) — в этом репозитории физически не может быть значений
  конкретного модпака.
- **`Pewdoloco/VanillaLauncher-localServer`** — остался модпак-репозиторием: `manifest.json`
  и моды по-прежнему публикуются туда через GitHub Releases (Admin-режим exe, собранного из
  этого репозитория и настроенного через экран "Настройки"), доки конкретного деплоя
  (`docs/SPEC.md`, `docs/ADMIN_GUIDE.md`, `docs/CLIENT_GUIDE.md`, `docs/OVERVIEW.md`) там же.
  Кода там больше нет.
- История ниже (весь этот `CHANGELOG.md`, `docs/TASKS.md`) — общая с тем репозиторием вплоть
  до этой точки; упоминания VanillaScary/Pewdoloco в ней — исторические (какой реальный
  деплой был референсом при разработке движка), не текущая конфигурация.

## Настройка лаунчера под модпак: автоопределение путей, экран "Настройки", модель движок/модпак

Реализация docs/TASK_PATH_AUTODETECT.md — задача расширилась от "автоопределение путей" до
"вообще все настройки, нужные, чтобы адаптировать один и тот же лаунчер под произвольный
модпак без правки кода". Полная сводка решений — в docs/ARCHITECTURE.md, раздел "Модули
(этап — настройка лаунчера под модпак)".

- **`PathAutoDetectService.TryFind`** — новый сервис в `VanillaLauncher.Client`: ищет прямую
  поддиректорию с совпадающим (регистронезависимо) именем в списке корней поиска. Используется
  и для клиентской папки (`MainWindow`, корни — `AppConfig.ClientSearchRoots`), и для серверной
  (`AdminWindow`, корень — всегда папка рядом с `.exe`, плюс `ServerSearchRoots`).
- **`SettingsWindow`** — новый экран, открывается только кнопкой внутри уже авторизованного
  `AdminWindow` (тот же гейт по паролю, что и весь Admin-режим; `MainWindow` про это окно не
  знает вообще). Все Admin-поля `AppConfig` редактируются: оба пути (с автопоиском и
  `OpenFolderDialog`), `ManifestUrl`, `GitHubOwner`/`GitHubRepo`, `ServerBatFileName`,
  `IncludeFolders`, `ServerExcludeMods`, `MaxBackupsToKeep`.
- **`AppConfig`** — новые поля `ClientFolderName`/`ClientSearchRoots`/`ServerFolderName`/
  `ServerSearchRoots`; новое свойство `IsConfigured` (`ManifestUrl`+`GitHubOwner`+`GitHubRepo`
  все заданы) — отличает "движок не адаптирован под модпак" от "у этого игрока просто другой
  путь". `Load()` больше не бросает исключение на пустых `ManifestUrl`/`ProfileRoot` — пустой
  конфиг теперь валидное состояние, а не ошибка чтения файла.
- **`appsettings.default.json`** — новый, по-настоящему пустой embedded-шаблон (без единого
  VanillaScary/Pewdoloco-значения), заменил собой прежнюю схему, где в эту роль (fallback при
  отсутствии внешнего файла) встраивался сам рабочий `appsettings.json` этого деплоя. Реальный
  `appsettings.json` (копируется рядом с exe при билде) дополнен новыми полями с реальными
  значениями VanillaScary (`ClientFolderName: "VanillaScary"`, `ServerFolderName: "Server VS"` и т.д.).
- **`.github/workflows/engine-release.yml`** — при теге `engine-v*` собирает self-contained
  exe и публикует его как GitHub Release **без** `appsettings.json` — код-часть требования
  "движок публикует свои релизы отдельно от модпаков"; фактическое разделение на отдельные
  GitHub-репозитории сознательно не делали в этой сессии (организационная миграция, не код).
- 8 новых тестов (`PathAutoDetectServiceTests`), `AppConfigTests` обновлены под новое поведение
  (`IsConfigured` вместо исключений) — 31 тест в `VanillaLauncher.Client.Tests`,
  37 в `VanillaLauncher.Admin.Tests`, все проходят.

## Фикс: appsettings.json не попал в релиз

Пользователь реально прошёл по docs/CLIENT_GUIDE.md — скачал `VanillaLauncher.exe` с
GitHub, положил в папку — и словил крах: "Не удалось загрузить appsettings.json:
Не найден файл конфигурации". В релизе `26.1.2-b1` был только exe — я загрузил его
отдельным ассетом (см. ниже) и забыл, что `appsettings.json` тоже нужен рядом; он не
встроен в exe, а просто лежит в папке публикации.

- **Немедленно:** `appsettings.json` из локальной publish-сборки залит в тот же релиз
  отдельным ассетом — https://github.com/Pewdoloco/VanillaLauncher-localServer/releases/download/26.1.2-b1/appsettings.json,
  разблокировало пользователя сразу.
- **По-настоящему:** `appsettings.json` теперь ещё и встроен в сборку `VanillaLauncher.Client`
  как `EmbeddedResource` (`LogicalName="VanillaLauncher.Client.appsettings.default.json"`).
  `AppConfig.Load()` при отсутствии внешнего файла читает встроенный дефолт вместо падения с
  `FileNotFoundException`. Теперь один exe работает сам по себе, даже если внешний
  `appsettings.json` забыли приложить к релизу или он потерялся — это в принципе не может
  повториться так же, как в первый раз.
- `VanillaLauncher.exe` в релизе `26.1.2-b1` пересобран и заменён (старый ассет удалён,
  залит новый — тот же самый бинарник, что перед этим проверен локально: чистая папка,
  только exe, без единого внешнего файла — не падает).
- 2 новых теста на `AppConfig.Load()` (fallback на встроенный дефолт; `Save()` после
  fallback-загрузки создаёт локальный файл) — 58 тестов всего.

## Первый настоящий релиз — 26.1.2-b1

Живой прогон "Опубликовать обновление" (нужно было настроить `VANILLALAUNCHER_GITHUB_TOKEN`
как переменную окружения — потребовался перезапуск процесса, унаследовавшего окружение).
Пайплайн отработал полностью и вскрыл реальную проблему.

- **Найдено:** до синхронизации `Server VS` держал 122 мода, после — 77 (точь-в-точь клиентская
  папка). `ServerFileSync` зеркалирует строго, включая удаление лишнего — 45 модов, которых не
  было в клиенте, с сервера пропали безвозвратно (бэкапа не нашлось). Один из оставшихся,
  `better_tab`, декларирует `"environment": "*"` в fabric.mod.json, но код трогает клиентский
  `net.minecraft.client.gui.screens.Screen` — сервер падал на старте с `NoClassDefFoundError`.
- **По согласованию с пользователем:** признали клиентскую папку (`D:\Modding\Servers\Client\VanillaScary`,
  новая единая структура клиент+сервер) источником истины вперёд, старый набор модов сервера
  восстанавливать было неоткуда.
- **Исправлено:** `ServerFileSync.MirrorAsync` получил `excludeFileNames` — список имён файлов,
  которые никогда не копируются (и удаляются, если уже были) при синхронизации на сервер.
  Не влияет на манифест/клиентскую раздачу — клиенты получают все моды как прежде.
  `AppConfig`/`appsettings.json`: новое поле `ServerExcludeMods`, заполнено по итогам разбора
  каждого мода в сборке — 15 честно помечены `"environment": "client"` в fabric.mod.json,
  `better_tab` добавлен по факту реального краша (метаданные мода недостоверны, только
  запуск вскрывает такое).
- **Проверено по-настоящему:** `Server VS` с исправленным набором (61 мод) запущен через
  реальную кнопку — `Done (1.236s)! For help, type "help"`, без ошибок, штатно остановлен.
  Свежий (не тот же процесс, что публиковал) клиент сверился с реальным манифестом
  `26.1.2-b1` на GitHub — `Сборка актуальна`, 171 файл, `Расхождений не найдено` (SHA-256
  каждого локального файла совпал с опубликованным). 3 новых теста на `ServerFileSync`
  (exclude: не копируется / удаляется если уже было / регистронезависимо) — 57 тестов всего
  (20 Client + 37 Admin).
- `docs/ADMIN_GUIDE.md`: добавлена процедура проверки нового мода на клиентскость перед
  публикацией (по `fabric.mod.json` + эмпирически через реальный запуск).

- `VanillaLauncher.exe` (publish-сборка, self-contained, ~68 МБ) загружен в тот же релиз как
  отдельный ассет через `GitHubReleaseClient` (переиспользован уже написанный и протестированный
  код, не вручную через сайт) — https://github.com/Pewdoloco/VanillaLauncher-localServer/releases/download/26.1.2-b1/VanillaLauncher.exe.
  Подтверждено прямым запросом (редирект на реальный файл с корректным именем).

Репозиторий: https://github.com/Pewdoloco/VanillaLauncher-localServer/releases/tag/26.1.2-b1

**Лаунчер полностью готов к раздаче друзьям** — скачивание, первый запуск, докачка модов,
подключение к серверу проверены на реальных данных от начала до конца.

## Подготовка к релизу

Пользователь спросил напрямую: "лаунчер готов к работе, можно скачать и запустить?" — честный ответ был "нет", и вот что нашлось и починено:

- **ProfileRoot был захардкожен под автора сборки.** `appsettings.json`, уезжающий внутрь exe, содержал путь `D:\Games\curseforge\Instances\VanillaScary` — папку конкретно этой машины. У любого другого человека такой папки нет. `MainWindow` теперь проверяет `Directory.Exists(ProfileRoot)` при старте; если папки нет — предлагает выбрать через нативный `Microsoft.Win32.OpenFolderDialog` (.NET 8 WPF), выбор сохраняется через новый `AppConfig.Save()` обратно в локальный `appsettings.json` — у каждого свой, ничего не коммитится.
- **Необработанные исключения роняли всё приложение с трассировкой на экране.** Починено: `MainWindow.InitializeAsync()` — try/catch вокруг загрузки конфига с понятным статусом; `App.xaml.cs` — глобальный `DispatcherUnhandledException`, показывает `MessageBox` вместо краша.
- **Не было ни одной publish-сборки.** Настроен `dotnet publish -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true` → один exe (`VanillaLauncher.exe`, ~70 МБ, self-contained). `RuntimeIdentifier`/`SelfContained` намеренно не в основной `PropertyGroup` csproj — иначе обычный `dotnet build` тоже стал бы self-contained win-x64 сборкой (другой путь выхода, сломало бы существующие тестовые скрипты, замедлило бы каждую итерацию). Проверено запуском реальной publish-сборки в изолированной от dev-окружения папке с намеренно несуществующим `ProfileRoot` — онбординг сработал корректно на настоящем артефакте, не только в Debug.
- `AppConfig`: добавлен `Save()` — перезаписывает файл, из которого конфиг был загружен, сохраняя остальные поля. 5 новых xUnit-тестов (`AppConfigTests`).
- `docs/CLIENT_GUIDE.md` и `docs/ADMIN_GUIDE.md` — краткие инструкции на русском: установка, кнопки, подключение к серверу (клиент); токен, пароль, публикация обновлений, публикация самого лаунчера (админ).
- `dotnet build` и `dotnet test` (20 Client + 34 Admin = 54) проходят без ошибок и предупреждений.

**Всё ещё не сделано:** реальный первый релиз на GitHub (`VANILLALAUNCHER_GITHUB_TOKEN` настраивается пользователем прямо сейчас) и публикация `VanillaLauncher.exe` как отдельного ассета, откуда друзья его скачают — см. docs/TASKS.md, раздел "Осталось до первого реального релиза".

## Этап 6 — Авторизация Admin-режима

- `src/VanillaLauncher.Admin/AdminAuthService.cs`: `SetPassword`/`VerifyPassword`/`HasPassword` — соль (16 байт) + PBKDF2-SHA256 (100 000 итераций), сравнение хешей через `CryptographicOperations.FixedTimeEquals` (защита от timing-атак). Plaintext-пароль нигде не сохраняется и не логируется.
- `SetAdminPasswordWindow` (первый запуск — два поля, пароль+подтверждение) и `AdminLoginWindow` (последующие запуски — одно поле, неверный пароль не закрывает окно) в `src/VanillaLauncher.Client.UI`.
- `MainWindow.AdminButton_Click`: гейт перед созданием `AdminWindow` — при первом запуске просит задать пароль, при последующих требует войти; `AdminWindow` не создаётся, пока `ShowDialog()` не вернёт `true`.
- Файл `admin-auth.json` живёт рядом с exe (build output, уже в `.gitignore` через `bin/`) и дополнительно явно прописан в `.gitignore` — не коммитится ни при каких обстоятельствах.
- "Привязка к ПК" из SPEC.md (опциональный пункт) не реализована — обоснование в ARCHITECTURE.md.
- `tests/VanillaLauncher.Admin.Tests`: 8 новых xUnit-тестов (`AdminAuthServiceTests`) — roundtrip верного/неверного пароля, персистентность между инстансами, файл никогда не содержит plaintext, разные соли для одинакового пароля.
- Сквозная проверка через живое UI Automation: первый запуск → установка пароля → `AdminWindow` открывается; перезапуск → неверный пароль отклонён (диалог остаётся открытым, `AdminWindow` не создаётся) → верный пароль пускает.
- `dotnet build` и `dotnet test` (15 Client + 34 Admin = 49) проходят без ошибок и предупреждений.

Не входило в объём Этапа 6 и не делалось: привязка пароля к конкретному ПК (см. ARCHITECTURE.md), ограничение числа попыток входа (не требовалось спецификацией).

## Этап 5 — Admin (публикация обновления одной кнопкой)

- `src/VanillaLauncher.Client/ManifestGenerator.cs`: генерация манифеста вынесена из `tools/GenerateManifest` в переиспользуемый код (модель `Manifest` и так жила в Client). URL для каждого файла строит вызывающая сторона через callback.
- Попутный фикс: GitHub Release assets не поддерживают "/" в имени — добавлен `ManifestGenerator.FlattenPathForAssetName` (плоские имена вида `mods_sodium.jar`) и флаг `--flat` в `tools/GenerateManifest` для той же схемы при ручной генерации. Без флага поведение инструмента не изменилось (обратная совместимость с Этапом 1).
- `src/VanillaLauncher.Admin/GitHubReleaseClient.cs` + `GitHubTokenProvider.cs`: создание релиза и загрузка ассетов через GitHub REST API. Токен — **только** из переменной окружения `VANILLALAUNCHER_GITHUB_TOKEN`, нигде не хранится в репозитории.
- `src/VanillaLauncher.Admin/ReleasePublisher.cs`: генерирует манифест с плоскими GitHub-совместимыми URL и заливает его + все файлы в новый релиз. Каждый релиз самодостаточен — ассеты старых релизов не переиспользуются (см. ARCHITECTURE.md, почему это осознанный выбор).
- `src/VanillaLauncher.Admin/ServerFileSync.cs`: зеркалирует mods/config из папки сборки (`ProfileRoot` — та же папка, что уже используется как источник для клиента) в серверную папку — копирует новое/изменённое по хешу, удаляет устаревшее.
