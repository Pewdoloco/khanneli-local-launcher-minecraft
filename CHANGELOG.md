# Changelog

## `engine-v1.6.0` — ручной контроль сервера, вотермарка версии релиза

- **Публикация больше не запускает сервер сама.** `PublishPipeline` был 5 шагов (бэкап →
  стоп → синхронизация → манифест → **старт сервера**) — последний шаг убран по запросу:
  админ должен явно нажать "Запустить сервер", когда сам решит, что готов (например, после
  проверки лога на NoClassDefFoundError у нового мода). Пайплайн теперь 4 шага, всегда
  оставляет сервер остановленным после публикации — не только при ошибке, как раньше.
- **Явное подтверждение остановки сервера.** «Остановить сервер» теперь показывает диалог
  "Сервер остановлен." при успехе — раньше единственная обратная связь была пассивным
  обновлением статуса, легко пропустить.
- **Закрытие лаунчера с запущенным сервером.** Раньше просто предупреждало и закрывалось.
  Теперь при попытке закрыть окно с запущенным сервером закрытие отменяется и предлагается
  сначала штатно остановить сервер — лаунчер закрывается только после реальной остановки
  (или сразу, если сервер и так не запущен).
- **Вотермарка последней опубликованной версии** в поле "Версия релиза" — `AdminWindow`
  спрашивает GitHub (`releases/latest`, без токена) при открытии окна и показывает тег серым
  текстом внутри пустого поля, чтобы не лезть на github.com каждый раз, вспоминая, что
  публиковалось прошлый раз.
- Уточнение по вопросу пользователя (не код): цифровая подпись exe от Windows SmartScreen
  технически возможна только через платный Authenticode-сертификат (EV — сразу снимает
  предупреждение, OV — набирает репутацию постепенно) — бесплатного/кодового решения нет.

119 тестов (81 Client + 38 Admin) — 2 теста `PublishPipelineTests` переименованы и
переписаны под новое поведение (сервер остаётся остановленным вместо перезапуска).

## `engine-v1.5.0` — доработки по фидбеку после `engine-v1.4.0`

- **Фикс: полная ссылка в GitHubOwner/EngineGitHubOwner ломала публикацию.** `Normalize`
  (для полей *Repo*) и новый эту версию нормализатор для *Owner* — это разные вещи: одна и
  та же вставленная ссылка `https://github.com/Owner/Repo` должна дать РАЗНЫЕ значения в
  двух разных полях (последний сегмент пути — Repo, предпоследний — Owner). Раньше поле
  GitHubRepo уже нормализовалось (`GitHubRepoNameNormalizer.Normalize`), но GitHubOwner —
  только `.Trim()`, поэтому вставленная в Owner ссылка сохранялась как есть и ломала
  публикацию 404-й. Новый `GitHubRepoNameNormalizer.NormalizeOwner` подключён и в Admin
  "Настройках" (`GitHubOwner`/`EngineGitHubOwner`), и в `ClientSetupWindow`.
- **Фикс: кнопки в MainWindow/AdminWindow могли выходить за границы окна.** Ряды кнопок
  были на `StackPanel` (не переносит по строкам) — после добавления кнопки "Настроить
  лаунчер" в `engine-v1.4.0` они перестали помещаться в окно шириной 640px. Заменено на
  `WrapPanel` — кнопки переносятся на новую строку вместо того, чтобы обрезаться/вылезать
  за край.
- **Копирование логов из консоли** — контекстное меню ("Копировать выделенное"/"Копировать
  всё") и `Ctrl+C` на списке журнала в `MainWindow` и `AdminWindow`. Раньше текст ошибок
  (особенно длинные JSON от GitHub API) нельзя было скопировать вообще — `ListBox` с
  `TextBlock`-шаблоном не даёт штатного выделения текста мышью.
- **"Водяной знак" в полях EngineGitHubOwner/EngineGitHubRepo** — показывает канонический
  дефолт (`Pewdoloco`/`khanneli-local-launcher-minecraft` из `EngineRepositoryDefaults`,
  добавлен в `engine-v1.4.0`) серым текстом внутри пустого поля, чтобы было видно, какое
  значение реально используется, не читая код/документацию.
- Уточнение по вопросу пользователя (не код, ответ в чате): имя переменной окружения для
  токена публикации — **фиксированное** `VANILLALAUNCHER_GITHUB_TOKEN`
  (`GitHubTokenProvider.EnvironmentVariableName`), не настраивается администратором. Если
  публикация нескольких разных модпак-репозиториев ведётся с одной машины — значение этой
  ОДНОЙ переменной нужно менять при переключении между репозиториями, а не заводить
  несколько по-разному названных переменных.

119 тестов (81 Client + 38 Admin) — 8 новых на `NormalizeOwner`.

## `engine-v1.4.0` — самостоятельная настройка клиента без Admin-режима

По запросу: раньше единственный способ раздать игрокам рабочий exe — это заранее
настроенная пара exe+appsettings.json (см. `engine-v1.3.0`/`docs/HANDOFF.md`), а если
GitHubOwner/GitHubRepo не заданы — единственный путь их вписать был через Admin-режим
(пароль, полные "Настройки"), что для игрока и избыточно, и концептуально не его роль.
GitHubOwner/GitHubRepo — не секрет (публичные координаты репозитория, не токен), поэтому
раздавать их через Admin-гейт не было необходимости.

- **`ClientSetupWindow`** — новое лёгкое окно, доступное из клиентского `MainWindow` без
  пароля (кнопка "Настроить лаунчер", видна только когда `!IsConfigured`). Просит только
  GitHubOwner + GitHubRepo (плюс опционально ClientFolderName), сам вычисляет ManifestUrl
  по стандартной формуле `.../releases/latest/download/manifest.json` и сохраняет —
  дальше сразу идёт обычная проверка обновлений. GitHubRepo нормализуется тем же
  `GitHubRepoNameNormalizer`, что и в Admin "Настройках" (полная ссылка тоже сработает).
  Полная "Настройки" (пароль, ServerDirectory, публикация и т.д.) не изменились и
  по-прежнему только в Admin-режиме.
- **`EngineRepositoryDefaults`** — канонический репозиторий движка (`Pewdoloco`/
  `khanneli-local-launcher-minecraft`) как фоллбек, когда EngineGitHubOwner/EngineGitHubRepo
  не заданы в конфиге. Раньше пустые поля просто отключали кнопку "Проверить обновление
  лаунчера" — теперь самообновление работает "из коробки" для любого модпака на этом
  движке без отдельной настройки этих двух полей. Явно заданное значение (например, у
  форка со своим движком) фоллбек не перезаписывает.
- Более подробные описания полей "Ожидаемое имя папки сборки/сервера" в Admin
  "Настройках" — раньше это была голая подпись без объяснения, что именно туда вписывать
  (только имя папки, не путь; прямая подпапка, не рекурсивно, без учёта регистра).

Live-прогон: собрал exe с нуля (без appsettings.json вообще), нажал "Настроить лаунчер",
вписал Owner + ссылку целиком вместо голого имени репозитория — нормализация сработала,
ManifestUrl посчитался верно, дальнейшая проверка обновлений прошла штатно (репозиторий
test-pack на момент прогона уже был удалён/приватный — 404 ожидаемый и не связан с этой
фичей).

109 тестов (без изменений — код UI-окна не покрыт юнит-тестами, как и остальные
Window.xaml.cs в этом репозитории; логика нормализации и формулы ManifestUrl уже
покрыта существующими `GitHubRepoNameNormalizerTests`).

## `engine-v1.3.1` — appsettings.json не пересоздавался после удаления

`AppConfig.Load()` при отсутствующем внешнем `appsettings.json` брал встроенный пустой
дефолт (`appsettings.default.json`) для работы в памяти, но никогда не писал его на диск —
файл появлялся только после первого явного `Save()` откуда-то из UI (например, после выбора
папки сборки). Из-за этого совет из `ClientGuide` ("удали appsettings.json и перезапусти —
он снова спросит папку") не работал: после удаления файла и перезапуска exe на диске
по-прежнему не было никакого файла, лаунчер просто каждый раз молча работал из
встроенного дефолта в памяти.

Теперь `Load()` сразу дописывает встроенный дефолт на диск, если внешнего файла не было —
best-effort (`try/catch` на `IOException`/`UnauthorizedAccessException`), чтобы не уронить
запуск в папке без прав на запись — та же причина, ради которой вообще есть этот фоллбек.
Существующий файл `Load()` по-прежнему не трогает.

2 новых теста в `AppConfigTests` (файл появляется на диске сразу после `Load()` без
отдельного `Save()`; существующий файл не перезаписывается) — 109 тестов всего
(71 Client + 38 Admin).

## `engine-v1.3.0` — живой прогон публикации, три реальных бага, обратная связь по UX

Первая настоящая живая публикация на "боевом" модпаке (VanillaScary/test-pack) после
разделения движок/модпак — нашла три реальных бага, которые не ловились юнит-тестами
(один сразу видел пользователь, два всплыли по ходу самого прогона).

- **`GitHubRepo`/`EngineGitHubRepo` = полный URL вместо имени репозитория** — причина
  исходной ошибки "GitHub API: не удалось выполнить создание релиза (404 NotFound)".
  Код строит `api.github.com/repos/{owner}/{repo}/...` из голого имени; если в поле вставили
  `https://github.com/Owner/Repo.git`, получался несуществующий путь. Добавлен
  `GitHubRepoNameNormalizer` — `SettingsWindow` теперь сам вырезает имя репозитория из
  вставленной ссылки при сохранении (убирает схему/хост/`.git`), независимо от того, что
  именно туда вставили.
- **GitHub переименовывает ассеты релиза при сохранении** (например, заменяет пробелы на
  точки: `Dangerous Fabric - 1.5.1.jar` → `Dangerous.Fabric.-.1.5.1.jar`) — `manifest.json`
  строил ссылку на файл заранее, из исходного имени, и она не совпадала с реальной, что
  давало 404 при скачивании клиентом (докачка при этом обрывалась целиком на первом
  же таком файле, дальше по списку ничего не скачивалось). `GitHubReleaseClient.UploadAssetAsync`
  теперь возвращает настоящий `browser_download_url` из ответа GitHub, и `ReleasePublisher`
  использует именно его — воспроизведено и проверено живым скачиванием 171 файла.
- **Ошибки GitHub API теперь с человеческой подсказкой** — `GitHubApiErrorTranslator`
  распознаёт частые случаи (404 — неверный репозиторий, 403 — токену не хватает прав,
  422 "Repository is empty" — в репозитории модпака нет ни одного коммита, 422
  "already exists" — версия уже опубликована) и добавляет строку "Подсказка: ..." поверх
  сырого сообщения GitHub. Подключено и к публикации, и к самообновлению движка.
- **Мод-пикер терял подчёркивания в именах файлов** — `CheckBox.Content` рендерится через
  `AccessText`, где одиночное "_" — маркер мнемоники ("_File" → подчёркнутая "F"), поэтому
  имена вроде `better_tab-...` отображались как `bettertab-...` — реальные данные это не
  портило (закрытие/выбор мода шёл по имени из замыкания, не из отображаемого текста), но
  сбивало с толку и глаз, и точный поиск элемента через UI Automation. Исправлено
  экранированием `_` → `__` только в отображаемом тексте.
- **Автоопределение client-only модов по метаданным** — при сканировании папки пикер теперь
  сам открывает каждый `.jar`, читает `"environment"` в `fabric.mod.json`
  (`FabricModEnvironmentReader`) и предотмечает моды с честным `"environment": "client"`.
  Не ловит моды, которые лгут в метаданных (`"environment": "*"`, но на деле клиентские,
  например `better_tab`) — для них по-прежнему нужен реальный запуск сервера и разбор лога.
- **Список консоли не переносил длинные строки** — `LogList` в `MainWindow`/`AdminWindow`
  показывал сообщения (особенно длинные JSON-ошибки GitHub) в одну строку с горизонтальной
  прокруткой. Добавлен `ItemTemplate` с `TextWrapping="Wrap"` и отключена горизонтальная
  прокрутка — текст переносится по ширине окна.
- **Инструкция разделена по ролям** — из клиентского окна (`MainWindow`) доступна только
  инструкция игрока, переключатель "Администратор" скрыт; из `AdminWindow` доступны обе
  роли. Раньше обе были видны отовсюду намеренно — решение пересмотрено по фидбеку.
- **Текст инструкции про то, где скачать exe, был устаревшим** — со времён, когда движок и
  модпак жили в одном репозитории. Поправлено: exe — из релизов репозитория ДВИЖКА
  (`engine-vX.Y.Z`), не репозитория модпака (там только контент).
- Формулировка описания `ServerExcludeMods` в "Настройках" уточнена — явно "список
  исключений", а не "итоговый список" (двусмысленно читалось как полный список всех модов).
- 25 новых тестов (`GitHubRepoNameNormalizerTests`, `FabricModEnvironmentReaderTests`,
  `GitHubApiErrorTranslatorTests`, плюс новые кейсы в `GitHubReleaseClientTests`/
  `ReleasePublisherTests` на переименование ассета GitHub) — 107 тестов всего
  (69 в `VanillaLauncher.Client.Tests`, 38 в `VanillaLauncher.Admin.Tests`).

## Доработки экрана "Настройки" и окна "Инструкция" по обратной связи

Пять точечных доработок после первого реального использования (полная сводка — в
docs/ARCHITECTURE.md, соответствующий раздел).

- **`GuideWindow`** — клиентская и админская инструкция объединены в одно окно с двумя
  переключателями (роль + краткое/полное, крупный шрифт у роли). Обнаружен и починен реальный
  баг: WPF `ShowDialog()` штатно отключает ВСЕ окна процесса (не только своего Owner) — любой
  модальный диалог где угодно в приложении временно блокировал открытую "Инструкцию".
  Починено перехватом `WM_ENABLE` в `GuideWindow` — проверено живьём, окно остаётся
  интерактивным (переключатель реально срабатывает), пока открыт диалог пароля администратора.
- **Поля руководств в "Настройках"** — read-only по умолчанию, кнопка "Редактировать"/"Готово"
  снимает защиту и увеличивает поле для удобной печати, обратно уменьшает при завершении.
- **`ClientSearchRoots`** — непустой generic-дефолт `%USERPROFILE%\curseforge\Instances`
  (был пустой список — автопоиск клиента не мог сработать без ручной настройки).
- **`MainWindow`**: кнопка "Сменить папку сборки" видна и активна всегда, не только в
  сценарии "путь не найден" — игрок может поменять путь сам в любой момент.
- **`SelectExcludedModsWindow`** — поиск по имени файла + постраничный список (10 строк на
  страницу, в духе библиотеки Steam) вместо одного сплошного скролла на 70-100+ модов.
  Отмеченное состояние не зависит от текущей страницы/фильтра.
- Более явные подписи `GitHubOwner`/`GitHubRepo` (репозиторий МОДПАКА) vs
  `EngineGitHubOwner`/`EngineGitHubRepo` (репозиторий ДВИЖКА) в "Настройках".
- Фикс валидации `MaxBackupsToKeep` — `WorldBackupService` требует `>= 1`, `SettingsWindow`
  до этого разрешал сохранить `0` (несостыковка проявлялась только при клике "Пересоздать мир").
- Новый тест на дефолт `ClientSearchRoots` — 46 тестов в `VanillaLauncher.Client.Tests`,
  37 в `VanillaLauncher.Admin.Tests`, все проходят.

## Самообновление exe движка

Раньше публикация модпака никогда не трогала сам `VanillaLauncher.exe` — новую версию
лаунчера можно было получить только вручную от админа. Теперь exe умеет проверять и
накатывать обновления сам (полная сводка — docs/ARCHITECTURE.md, раздел "Самообновление exe
движка").

- **`EngineVersion.Current`** — константа с версией этой сборки движка; CI
  (`.github/workflows/engine-release.yml`) сверяет её с тегом `engine-vX.Y.Z` при публикации
  и проваливает релиз при расхождении.
- **`EngineSelfUpdater`** — `CheckForUpdateAsync` (GitHub Releases API, сравнение версий через
  `System.Version`, без токена) и `PrepareAndLaunchUpdateAsync` (скачивает новый exe, готовит
  и запускает `.bat`, который дожидается завершения текущего процесса по PID, подменяет файл
  и перезапускает).
- **`AppConfig.EngineGitHubOwner`/`EngineGitHubRepo`** — отдельно от `GitHubOwner`/`GitHubRepo`
  (те про репозиторий модпака с контентом) — репозиторий, откуда качать новую версию exe.
- **`MainWindow`**: кнопки "Проверить обновление лаунчера"/"Обновить лаунчер" — по аналогии
  с существующей парой для контента модпака. Осознанно в клиентском окне, а не только в
  Admin — целевая аудитория в первую очередь игроки со старыми копиями exe.
- 7 новых тестов на `EngineSelfUpdater.CheckForUpdateAsync` (новее/такая же/старее версия,
  тег без префикса `engine-v`, релиз без нужного ассета, HTTP-ошибка, User-Agent заголовок) +
  2 на `AppConfig` (дефолты и сохранение `EngineGitHubOwner`/`Repo`) — 82 теста всего.
- Сознательно не сделано: проверка целостности скачанного exe (SHA-256/подпись) — TLS
  обеспечивает подлинность канала загрузки, отдельной сверки хеша нет.

## Переменные окружения в путях поиска, выбор модов из папки, встроенная инструкция

Три доработки экрана "Настройки" по итогам живого использования уже разделённого
движка (полная сводка решений — в docs/ARCHITECTURE.md, соответствующий раздел).

- **`%VAR%` в `ClientSearchRoots`/`ServerSearchRoots`** — `PathAutoDetectService.TryFind`
  раскрывает переменные окружения (`%USERPROFILE%` и т.д.) перед проверкой каждого корня.
  Раньше корень поиска приходилось задавать абсолютным путём одного конкретного человека —
  не работало у друзей с другим именем пользователя/диском.
- **`SelectExcludedModsWindow`** — кнопка в "Настройках" сканирует `*.jar` в папке (по
  умолчанию `{ProfileRoot}\mods`) и даёт отметить `ServerExcludeMods` чекбоксами вместо
  ручного набора имён файлов текстом. Мёржит с уже существующими записями, не относящимися
  к просканированной папке, — ничего не стирает случайно.
- **`GuideWindow`** — немодальное окно "Инструкция" (`Show()`, не `ShowDialog()`), доступно
  из `MainWindow` (клиентский текст) и `AdminWindow` (админский), с переключением
  Краткое/Полное. Текст — новые поля `AppConfig` (`ClientGuideShort/Full`,
  `AdminGuideShort/Full`) с непустыми generic-дефолтами (`DefaultGuides`), редактируется
  под конкретный модпак через "Настройки".
- Ручная генерация `manifest.json` (`tools/GenerateManifest`) задокументирована в
  `docs/ADMIN_GUIDE.md` репозитория модпака — раньше существовала только в коде.

Продолжение предыдущего пункта (модель "движок/модпак" была реализована на уровне кода, но

- **`Pewdoloco/VanillaLauncher-localServer`** — остался модпак-репозиторием: `manifest.json`

## Фикс: appsettings.json не попал в релиз

Пользователь реально прошёл по docs/CLIENT_GUIDE.md — скачал `VanillaLauncher.exe` с
GitHub, положил в папку — и словил крах: "Не удалось загрузить appsettings.json:
Не найден файл конфигурации". В релизе `26.1.2-b1` был только exe — я загрузил его
отдельным ассетом (см. ниже) и забыл, что `appsettings.json` тоже нужен рядом; он не
встроен в exe, а просто лежит в папке публикации.

- **Немедленно:** `appsettings.json` из локальной publish-сборки залит в тот же релиз
  отдельным ассетом — https://github.com/Pewdoloco/VanillaLauncher-localServer/releases/download/26.1.2-b1/appsettings.json,
  разблокировало пользователя сразу.
- **По-настоящему:** `appsettings.json` теперь ещё и встроен в сборку `VanillaLauncher.Client`
  как `EmbeddedResource` (`LogicalName="VanillaLauncher.Client.appsettings.default.json"`).
  `AppConfig.Load()` при отсутствии внешнего файла читает встроенный дефолт вместо падения с
  `FileNotFoundException`. Теперь один exe работает сам по себе, даже если внешний
  `appsettings.json` забыли приложить к релизу или он потерялся — это в принципе не может
  повториться так же, как в первый раз.
- `VanillaLauncher.exe` в релизе `26.1.2-b1` пересобран и заменён (старый ассет удалён,
  залит новый — тот же самый бинарник, что перед этим проверен локально: чистая папка,
  только exe, без единого внешнего файла — не падает).
- 2 новых теста на `AppConfig.Load()` (fallback на встроенный дефолт; `Save()` после
  fallback-загрузки создаёт локальный файл) — 58 тестов всего.

## Подготовка к релизу

Пользователь спросил напрямую: "лаунчер готов к работе, можно скачать и запустить?" — честный ответ был "нет", и вот что нашлось и починено:

- **ProfileRoot был захардкожен под автора сборки.** `appsettings.json`, уезжающий внутрь exe, содержал путь `D:\Games\curseforge\Instances\VanillaScary` — папку конкретно этой машины. У любого другого человека такой папки нет. `MainWindow` теперь проверяет `Directory.Exists(ProfileRoot)` при старте; если папки нет — предлагает выбрать через нативный `Microsoft.Win32.OpenFolderDialog` (.NET 8 WPF), выбор сохраняется через новый `AppConfig.Save()` обратно в локальный `appsettings.json` — у каждого свой, ничего не коммитится.
- **Необработанные исключения роняли всё приложение с трассировкой на экране.** Починено: `MainWindow.InitializeAsync()` — try/catch вокруг загрузки конфига с понятным статусом; `App.xaml.cs` — глобальный `DispatcherUnhandledException`, показывает `MessageBox` вместо краша.
- **Не было ни одной publish-сборки.** Настроен `dotnet publish -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true` → один exe (`VanillaLauncher.exe`, ~70 МБ, self-contained). `RuntimeIdentifier`/`SelfContained` намеренно не в основной `PropertyGroup` csproj — иначе обычный `dotnet build` тоже стал бы self-contained win-x64 сборкой (другой путь выхода, сломало бы существующие тестовые скрипты, замедлило бы каждую итерацию). Проверено запуском реальной publish-сборки в изолированной от dev-окружения папке с намеренно несуществующим `ProfileRoot` — онбординг сработал корректно на настоящем артефакте, не только в Debug.
- `AppConfig`: добавлен `Save()` — перезаписывает файл, из которого конфиг был загружен, сохраняя остальные поля. 5 новых xUnit-тестов (`AppConfigTests`).
- `docs/CLIENT_GUIDE.md` и `docs/ADMIN_GUIDE.md` — краткие инструкции на русском: установка, кнопки, подключение к серверу (клиент); токен, пароль, публикация обновлений, публикация самого лаунчера (админ).
- `dotnet build` и `dotnet test` (20 Client + 34 Admin = 54) проходят без ошибок и предупреждений.

**Всё ещё не сделано:** реальный первый релиз на GitHub (`VANILLALAUNCHER_GITHUB_TOKEN` настраивается пользователем прямо сейчас) и публикация `VanillaLauncher.exe` как отдельного ассета, откуда друзья его скачают — см. docs/TASKS.md, раздел "Осталось до первого реального релиза".
